# Accessibility Audit — 2026-05-04

Scope: every major Slothing route — marketing home, dashboard, bank, studio,
opportunities, opportunities/[id], settings, profile, analytics, calendar,
ats-scanner. The goal of this pass is "0 critical/serious axe-core violations"
on each audited route, plus targeted fixes for issues that axe can't catch
automatically (focus management, drawer semantics, kanban affordances).

The route-by-route axe scan is wired into `e2e/accessibility.spec.ts` and runs
under `npm run test:e2e`. The harness asserts zero `critical` or `serious`
violations on every route in `AUDIT_ROUTES`, with third-party portal nodes
(Next.js error overlay, NextAuth auth UI) excluded.

---

## How to reproduce

```bash
npm run test:e2e -- e2e/accessibility.spec.ts
```

Each route runs through axe-core 4.x using the WCAG 2.1 A + AA rule set
(`wcag2a`, `wcag2aa`, `wcag21a`, `wcag21aa`). When violations are found, the
test logs the rule id, impact, help text, and node count so the failure is
actionable in CI.

## Findings & fixes

### A1 — Nested `<main>` landmarks in `/studio` and `/opportunities`
**Severity:** serious (axe `landmark-no-duplicate-main`).

`src/app/(app)/layout.tsx` already exposes a single `<main id="main-content">`
landmark. Both `/studio` and `/opportunities` rendered an inner `<main>` inside
that landmark, producing two `main` regions per route.

- `src/app/(app)/studio/page.tsx`: the preview tabpanel was a `<main>` element.
  It is functionally a tabpanel, so it has been switched to `<section>` with
  the same `role="tabpanel"` plumbing intact (`aria-labelledby`,
  `data-document-editor-root`, etc.).
- `src/app/(app)/opportunities/page.tsx`: the kanban/list content area was a
  bare `<main>`. It is now a `<section aria-label="Opportunities list">`.

### A2 — Sidebar drawer lacked dialog semantics & focus management
**Severity:** serious (focus order + `aria-modal`).

The sidebar (`src/components/layout/sidebar.tsx`) is a slide-in drawer on
mobile. Previously it had no dialog semantics, no programmatic focus on open,
and the underlying page was still scrollable.

Fixes:
- The mobile open button now declares `aria-haspopup="dialog"`,
  `aria-expanded`, and `aria-controls="primary-sidebar"`.
- The `<aside id="primary-sidebar">` adds `role="dialog"` and
  `aria-modal="true"` only when the drawer is open (so the desktop
  `complementary` landmark is preserved at ≥1024px).
- On open, focus is moved to the close button; on close (transition
  true→false), focus is returned to the trigger button so keyboard users
  land back where they started.
- Tab is trapped inside the drawer while it is open: tabbing past the last
  focusable element wraps to the first, and shift+tab from the first wraps
  to the last. Combined with `aria-modal`, escape-to-close, and the scroll
  lock, this gives the drawer the standard modal interaction model.
- Body scroll is locked while the drawer is open and restored on close.

### A3 — Avatar `<img>` carried empty alt text
`src/app/(app)/profile/page.tsx` rendered the user's avatar with `alt=""`,
which is technically valid as decorative but caused screen readers to read
nothing where an identity affordance is otherwise present. The alt now
resolves to `"<name> profile photo"` (falls back to `"Profile photo"`).

### A4 — Kanban boards had no announced region or scroll affordance
**Severity:** moderate (perception of `min-w-[1480px]` overflow on tablet).

Both `OpportunityKanbanView` (in `src/app/(app)/opportunities/page.tsx`) and
`JobKanbanView` (`src/components/jobs/job-kanban-view.tsx`) wrap a fixed-width
grid in `overflow-x-auto`. Keyboard users could not focus the scroll
container and screen readers had no hint that the region was a kanban board.

The wrapper now declares:

```tsx
role="region"
aria-label="Opportunity kanban board"
aria-roledescription="kanban board"
tabIndex={0}
```

This makes the scroll container keyboard-focusable (so arrow keys scroll the
overflow) and labels it for assistive tech. The columns inside are still
`<section aria-label="…">` with `<h2>` titles.

### A5 — Skip-to-main link, landmarks, headings
Already in place before this pass and verified during the audit:

- `src/app/(app)/layout.tsx` ships a working **Skip to main content** link as
  the first focusable element, plus `<main id="main-content">` and the
  `<aside aria-label="Main navigation">` landmark.
- The marketing layout uses `<header>` (Navbar), `<main>`, `<footer>`
  landmarks.
- Headings on each audited page start at `h1` and step monotonically
  (verified spot-checks: dashboard, opportunities, calendar, profile,
  studio header).
- Icon-only buttons across the app (sidebar collapse tooltips, notification
  controls, Studio panel toggles, calendar month nav, jobs toolbar clear) all
  carry `aria-label`. The new `evaluateTouchTarget` helper formalises the
  44 px comfort target the codebase already uses with `min-h-11`.

### A6 — Color contrast
The T3 design-token migration (semantic CSS variables for foreground / muted
/ destructive / success / warning / info) means contrast is governed by
`globals.css`. Spot-checked against axe-core: no `color-contrast` violations
on the audited routes.

The calendar legend, dashboard "Recent activity" indicators, and analytics
status pills all back color cues with adjacent text labels, so meaning is
not conveyed by color alone (WCAG 1.4.1).

### A7 — Form labels
Every visible form control on the audited routes has either a `<Label
htmlFor="…">` (Studio bank entry inputs, Opportunities filters, Settings
panels) or an explicit `aria-label` (Opportunities search, sort select,
clear-search buttons). This was verified by walking the code in
`src/app/(app)/opportunities/page.tsx`, `src/app/(app)/profile/page.tsx`,
`src/app/(app)/settings/`, and `src/components/jobs/jobs-toolbar.tsx`.

## Helpers added

`src/lib/a11y.ts` introduces small utilities used both at runtime and by the
audit harness:

- `evaluateTouchTarget(box)` — returns `pass | comfort-warning | fail` against
  the WCAG 2.5.8 floor (24 px) and the comfort target (44 px) the codebase
  uses for tap surfaces.
- `parseHeadingLevel(tag)` — turns `H1`–`H6` (any case) into a numeric level.
- `findHeadingOrderIssues(headings)` — flags skipped levels (e.g. h2 → h4).
- `hasAccessibleName(inputs)` — encapsulates the
  `aria-label / aria-labelledby / textContent / title` precedence used by the
  Playwright "interactive elements have accessible names" test.

All four are unit-tested in `src/lib/a11y.test.ts`.

## Acceptance

- Every route in `AUDIT_ROUTES` (`e2e/accessibility.spec.ts`) is asserted to
  have zero critical/serious axe violations.
- Heading order, skip link, landmarks, alt text, and form labelling were
  hand-verified for each route as documented above.
- The remaining "minor" axe rules (best practices) are tracked but not
  blocking: they primarily flag `<svg>` decorative elements that are already
  `aria-hidden`.
