# Public-launch trust follow-ups — 2026-05-12

Generated by the public-launch security/privacy trust audit. Companion document to `docs/security-audit-2026-05-04.md` (backend / OWASP).

The audit fixed misleading user-facing copy across the extension marketing page, Privacy Policy, Terms of Service, and ATS scanner SEO. The items below are out-of-scope (require legal review, broader refactors, or product decisions) and should be tracked as separate work before public launch.

## High priority — block public launch / paid launch

### 1. Terms of Service: jurisdiction + dispute-resolution [TBD]

- `apps/web/src/app/[locale]/terms/page.tsx` line ~200, ~221.
- Currently published at `/terms` with literal `[JURISDICTION TBD before public launch]` and `[DISPUTE RESOLUTION TBD before public launch]` placeholders.
- A "Pre-launch draft" banner was added at the top as a stopgap. Real fix:
  - Get legal counsel to fill jurisdiction, arbitration provider/forum, class-action waiver, small-claims exception, and any state-specific carve-outs.
  - Remove the banner.
  - Add a data processing addendum (DPA), subprocessor list, and a cookie consent mechanism for EU/UK visitors.

### 2. Google OAuth: move from all-upfront to incremental authorization

- `apps/web/src/auth.config.ts` lines 5–14.
- Every Google sign-in currently requests `gmail.readonly`, `gmail.send`, `calendar`, `drive.file`, `contacts.readonly` together. Users who never use those features still grant them at sign-in.
- This forces over-consent, raises the Google OAuth verification bar (sensitive scopes), and contradicts data-minimization.
- Fix: at sign-in request only `openid email profile`. Request additional scopes only when the user opts into Calendar/Drive/Gmail/Contacts features. Touch points:
  - `auth.config.ts` — strip scope list.
  - `apps/web/src/app/[locale]/(app)/settings/google/` (or equivalent) — add per-feature connect buttons that re-prompt with the needed scope.
  - `apps/web/src/lib/google/` — guard each API call by a scope check; surface a "connect Gmail" CTA when missing.
- After this lands, the Privacy Policy disclosure added in this audit can be reverted to "we request scopes when you connect each feature".

### 3. Extension auto-tracking default needs a clearer consent moment

- `apps/extension/src/shared/types.ts:214` — `autoTrackApplicationsEnabled: true` (default).
- `apps/extension/src/content/tracking/submit-watcher.ts` — on form submit on a supported job site, the extension auto-creates an `applied` opportunity (and a screenshot if `captureScreenshotEnabled` is on; default off).
- Neither the marketing page nor the Privacy Policy explicitly tells users that submitting a job form will be tracked and sent to Slothing.
- Pick one:
  - Flip default to `false` and add a first-submission opt-in toast.
  - Or keep default `true` but show an explicit disclosure on first sign-in (and add a line to the extension Privacy & trust section of `apps/web/src/app/[locale]/(marketing)/extension/page.tsx`).

## Medium priority

### 4. Sanitize server-side error stacks before logging

- ~20 server routes log full `error.stack`. Examples:
  - `apps/web/src/app/api/upload/route.ts:183,272,338,355,383`
  - `apps/web/src/app/api/cover-letter/generate/route.ts:228`
  - `apps/web/src/app/api/tailor/route.ts:250`
  - `apps/web/src/app/api/parse/route.ts:127,144`
  - `apps/web/src/app/api/resume/generate/route.ts:87`
  - …and 10 more under `/api/`.
- Stacks can include frames with embedded user content when libraries surface that in error messages. Not exposed to the client (the routes already return generic messages), but is retained by the host's log aggregation.
- Add `src/lib/log/safe-error.ts` exporting `safeErrorLog(prefix, err)` that strips `error.stack` by default and only includes it when `SLOTHING_LOG_STACKS=1`. Replace all 20 sites.

### 6. Extension token revocation UX

- `extension_sessions` table already exists, but the web app has no UI for users to see and revoke connected extensions.
- If a user loses a device with Columbus installed, today they can only rotate by signing the extension out (which requires the device).
- Add `/settings/connected-devices` listing extension session ID, last-seen UA / region, and a "Revoke" button that invalidates the token server-side.

## What was verified accurate (no changes needed)

- **ATS scanner**: `/api/scanner/parse-resume` does NOT persist resumes, does NOT call LLMs, and explicitly strips `rawText` from its response. Marketing claim ("parsed on our servers, not saved, not shared") is honest.
- **Multi-user data isolation**: every API route scopes by `user_id` after the 2026-05-04 audit. No regressions found.
- **CSP / security headers**: middleware emits a strong default-set; nothing to fix here.
- **Auth-disabled posture**: `requireAuth()` returns 503 when NextAuth is not configured, unless the explicit `SLOTHING_ALLOW_UNAUTHED_DEV=1` dev bypass is set and `NODE_ENV !== "production"`. Safe.
