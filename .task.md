# [JOBS] /en/applications route is 404 - UX leak: dashboard + nav reference applications, no destination

## Problem

There is no `apps/web/src/app/[locale]/(app)/applications/` directory. The route `/en/applications` returns Next.js's default 404. However:

- The product talks about "applications" pervasively (`pipeline.applied`, `analytics.applicationsSent`, `applications/recent`, etc.)
- The dashboard's "Pipeline" panel groups stages by status: `saved | applied | interviewing | offered`. A user clicking "Applied (3)" reasonably expects to land on an applications list.
- `/api/applications` also doesn't exist - the data layer uses `/api/jobs` (which is itself a deprecation stub that 410s, see `apps/web/src/app/api/jobs/route.ts` and `deprecated.ts`) plus `/api/opportunities`.

Net effect: a first-time user who learned the word "applications" from the marketing landing page or dashboard pipeline has no entry point. The current implementation conflates "applications" with "opportunities" but doesn't tell the user.

## Repro

1. `pnpm dev`
2. `curl -s -o /dev/null -w "%{http_code}\n" localhost:3000/en/applications` -> `404`
3. `curl -s -o /dev/null -w "%{http_code}\n" localhost:3000/en/jobs` -> 307 (redirects to `/en/opportunities` - fine, has compat shim)
4. `curl -s -o /dev/null -w "%{http_code}\n" localhost:3000/api/applications` -> `404`
5. Open dashboard -> click "Applied" stage in pipeline -> goes to `/opportunities` (good) but with no filter applied -> user has to figure out "applied" = a status filter, not a separate page.

## Suspected cause

Mid-rename: project renamed `/jobs` -> `/opportunities` (compat shim in `(app)/jobs/page.tsx`), but never built an `applications` route because internally applications and opportunities share a model (jobs table, `status` column).

## Fix

Pick one of two paths - recommend **A**:

**A. Make `/en/applications` a thin alias that opens `/en/opportunities?status=applied,interviewing,offered`.**
- Mirror `(app)/jobs/page.tsx` pattern: server-side redirect with `searchParams` carrying the filter.
- Update dashboard `PipelineSummary` stage links to deep-link into the filtered view (`/opportunities?status=applied`, etc.) so the click actually narrows the list.

**B. Build a real applications view at `/en/applications` showing only rows where `status IN ('applied','interviewing','offered','rejected')`, with submission timestamps surfaced.**
- More work, but matches user mental model.
- Reuse `OpportunitiesList` with a baked-in status filter.

Either way: also ensure `/api/applications` returns a 307 -> `/api/opportunities?status=...` (or implement a filtered handler), so the URL is stable for the extension and for tests.

## Acceptance criteria

- [ ] `curl -sI localhost:3000/en/applications` returns 200 (or 307 to a working URL), not 404.
- [ ] Dashboard pipeline stage chips (Saved / Applied / Interviewing / Offered) deep-link with the right `status` query param.
- [ ] `OpportunitiesList` honors `searchParams.status` when present.
- [ ] Snapshot test for the new route or the new filter behavior.
- [ ] No regression for `/en/opportunities` without filters.
