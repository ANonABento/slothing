# [AUTH] /en/sign-in redirects to /dashboard when NextAuth disabled — dead-end UX, no operator signal

## Problem

`apps/web/src/app/[locale]/(auth)/sign-in/page.tsx:17-20` does:

```ts
if (!isNextAuthConfigured()) {
  redirect(`/${params.locale}/dashboard`);
}
```

When auth is disabled (e.g. `pnpm dev` without `.env`), hitting `/en/sign-in` 308-redirects to `/en/dashboard` with zero feedback. A new contributor or a deploy operator who's debugging "why is the sign-in flow broken?" gets:
- No error message
- No banner saying "auth is disabled in this environment"
- No link explaining what envs are missing

It's the worst kind of silent UX: the page that says "Sign In" sends you somewhere that requires being signed in. If `(app)/layout.tsx`'s dev-bypass (see related `[AUTH]` ticket) ever gets fixed to redirect-on-no-session, this becomes an infinite loop.

## Repro

1. Confirm `.env` has no `GOOGLE_CLIENT_ID`.
2. `pnpm dev`
3. `curl -sI localhost:3000/en/sign-in` → `307 Temporary Redirect` → `/en/dashboard`
4. No log line, no banner, nothing tells you sign-in is intentionally disabled.

## Suspected cause

The redirect was added as a dev-only escape hatch so the marketing landing page's "Sign in" CTA wouldn't error out when env vars are missing. Quick fix, wrong long-term shape.

## Fix

Replace the silent redirect with an explicit "Auth disabled" page state:

1. In `sign-in/page.tsx`, when `!isNextAuthConfigured()`, render a dedicated component (`<AuthDisabledCard />`) instead of redirecting. The card should show:
   - Heading: "Sign-in is disabled in this environment"
   - Body: "This Slothing instance is running without Google OAuth credentials. Missing: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `NEXTAUTH_SECRET`. See `.env.example`."
   - In dev (`NODE_ENV !== "production"`): a button "Continue to dashboard (dev mode)" — the only way to reach `/dashboard` without configured auth, explicit user action.
   - In prod: no such button. Just "Contact your administrator."
2. Show the same card if the user lands here without a session AND auth IS configured (current implicit path leaks through `(app)` layout's redirect into sign-in).
3. Add Playwright test: with auth disabled, `/en/sign-in` returns 200 with the disabled card, NOT a 307.

## Acceptance criteria

- [ ] `curl -sI localhost:3000/en/sign-in` returns 200, not 307, when auth is disabled.
- [ ] Page renders the disabled-state card with the missing env list.
- [ ] In dev mode only, "Continue to dashboard (dev mode)" link works.
- [ ] In prod build with auth disabled, no dev-mode link is rendered (gated by `process.env.NODE_ENV !== "production"`).
- [ ] No regression for the configured-auth flow (Google + magic link still work).
- [ ] Snapshot test added for the disabled state.
