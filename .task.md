# Stop env-validation warnings from spamming dev log on every request

## Problem

The dev log prints these two warnings on **every** request, not once at boot:

```
[env] NextAuth (Google) disabled — missing: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, NEXTAUTH_SECRET. See .env.example for the correct variable names.
[env] LLM Providers disabled — missing one of: OPENAI_API_KEY, ANTHROPIC_API_KEY, OPENROUTER_API_KEY, OLLAMA_BASE_URL. See .env.example for the correct variable names.
```

The code in `apps/web/src/lib/env.ts:130-145` already tries to dedupe via a `globalThis.__slothingEnvValidationLogged` flag inside `ensureEnvValidated()` — but the flag doesn't survive whatever Next 14 dev does between requests, so the warnings still spam.

## Repro

1. `cd /home/anonabento/slothing && pnpm dev`
2. `curl localhost:3000/en` three times
3. Tail `/tmp/slothing-dev.log` — both warnings appear **3×** (once per request) instead of once total

## Root cause

`ensureEnvValidated()` is currently called from `apps/web/src/app/layout.tsx:11` at module-eval time. In Next 14 dev mode, the RSC layout module is re-evaluated frequently enough that the `globalThis` dedupe flag is ineffective.

## Fix

Move the env-validation call into **`apps/web/instrumentation.ts`** — Next.js's official one-time-init hook (runs once per server process, never per request). Specifically:

1. Create `apps/web/instrumentation.ts` (if it doesn't exist) — Next auto-picks it up.
2. Export `async function register()` that calls `logEnvValidation()` (the un-deduped version is fine since instrumentation runs once).
3. Remove the `ensureEnvValidated()` call from `apps/web/src/app/layout.tsx:11`.
4. Remove the now-unused `ensureEnvValidated()` function from `src/lib/env.ts` (or keep it for now — your call).
5. Make sure `next.config.*` does NOT have `experimental.instrumentationHook: false` (in Next 14 it's opt-in via `experimental.instrumentationHook: true` until 14.x; in 15+ it's on by default — check version and gate accordingly).
