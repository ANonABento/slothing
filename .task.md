# [i18n] Root <html lang="en"> hardcoded â€” non-English locales serve wrong lang attribute

## Problem

`apps/web/src/app/layout.tsx:18-19` hardcodes `<html lang="en" ...>` for **every** locale. Users on `/ja`, `/ko`, `/zh-CN`, `/hi`, etc. all receive `lang="en"` in the document root. This breaks:

- Screen readers (announce content in wrong language voice/pronunciation)
- Browser translation prompts (Chrome offers to "translate from English" on Japanese pages)
- SEO â€” Google uses `lang` as a signal; mismatched lang vs content hurts ranking in non-English markets
- CSS `:lang()` selectors and font fallback chains

## Repro

```sh
grep -n 'lang=' apps/web/src/app/layout.tsx
# 18:    <html
# 19:      lang="en"
```

Confirms the root layout is locale-agnostic. The locale layout at `apps/web/src/app/[locale]/layout.tsx` does NOT (and cannot) override the `<html>` tag â€” only the root layout owns it.

## Root cause

`apps/web/src/app/layout.tsx` is the **root** layout (above `[locale]`), so it runs without locale context. The current Next.js App Router pattern for next-intl is to either:
1. Move the `<html>` to `[locale]/layout.tsx` and delete `app/layout.tsx`'s `<html>`, or
2. Read the locale from headers/cookies in the root layout (works server-side).

The repo went with neither â€” `<html lang="en">` is hardcoded.

## Fix

Move the `<html>` element into `apps/web/src/app/[locale]/layout.tsx` so each locale renders its own root. Concretely:

1. In `apps/web/src/app/[locale]/layout.tsx`:
   - Import `getMessages` from `next-intl/server` and the existing locale check.
   - Render `<html lang={params.locale} dir={isRtl(params.locale) ? "rtl" : "ltr"}>` (none of our 8 locales are RTL, but add a stub helper for future Arabic/Hebrew support).
   - Move `<head>` + theme preload script + `<body>` from `app/layout.tsx` into here.
2. In `apps/web/src/app/layout.tsx`:
   - Reduce to a minimal pass-through that returns `children` directly (no `<html>` / `<body>`). Or, if Next requires a root html, render `<html lang="en"><body>{children}</body></html>` ONLY as a fallback for the `not-found` route and accept the duplication carefully.
3. Add `apps/web/src/i18n/dir.ts` exporting `function localeDir(locale: AppLocale): "ltr" | "rtl"` that currently returns `"ltr"` for all 8 locales. Wire it into the `<html dir>` attribute.
