# [JOBS] DB layer defaults userId to "default" in 60+ functions — silent cross-tenant leak risk

## Problem

Across `apps/web/src/lib/db/`, at least 60 query functions have a default parameter `userId: string = "default"`:

```
custom-templates.ts: 5 functions
interviews.ts:       7 functions
analytics-queries.ts: 6 functions
jobs.ts:             9 functions
company-research.ts: 5 functions
opportunity-contacts.ts: 3 functions
cover-letters.ts:    4 functions
... (and more)
```

The intent was a "local dev convenience" so test scripts can call `getJobs()` with no args. But this means **any caller that forgets to thread `userId` through silently reads/writes the `"default"` user's rows**. There is no compile-time signal (no `null`-or-userId, no branded type, nothing). A future refactor that adds a new API route and forgets one `userId` argument leaks data — and TypeScript won't catch it because `"default"` is a valid string.

Combined with the [AUTH] ticket (when NextAuth is disabled, `requireAuth()` returns `userId="default"`), the entire local dev environment runs on one shared row. In prod, if any single API route forgets to pass `userId` from `authResult.userId`, it'll silently query the dev row.

## Repro

1. `grep -rn 'userId.*=.*"default"' apps/web/src/lib/db/` → 60+ matches.
2. Search call sites that don't pass `userId`:
   ```
   rg 'getJobs\(\)|getInterviews\(\)|getCustomTemplates\(\)' apps/web/src/
   ```
3. Any match is a latent leak.

## Suspected cause

Early-stage code: the project started as single-tenant, multi-tenant was bolted on, and the defaults were never removed because tests rely on them.

## Fix

1. **Remove the default parameter values.** Make `userId: string` required everywhere in `lib/db/`.
2. Fix test fixtures: tests that previously called `getJobs()` should call `getJobs("test-user")` (or use a shared `TEST_USER_ID` constant).
3. Add an ESLint rule (`no-restricted-syntax` with selector `Parameter[name.name='userId'][typeAnnotation.typeAnnotation.type='TSStringKeyword'][optional!=true] > AssignmentPattern`) that disallows default values for `userId` parameters anywhere in the codebase.
4. Audit every call site of these DB functions. Anything not passing an explicit `userId` should fail typecheck after the change; fix each one to thread the value from `authResult.userId` (or the test fixture).

## Acceptance criteria

- [ ] `grep -rn 'userId.*=.*"default"' apps/web/src/lib/db/` returns 0 matches.
- [ ] `pnpm type-check` passes after the change with no `// @ts-expect-error` workarounds.
- [ ] All `apps/web/src/lib/db/*.test.ts` files updated to pass explicit user IDs.
- [ ] All API route handlers in `apps/web/src/app/api/` thread `authResult.userId` to every DB call.
- [ ] ESLint rule in place; CI fails if a new `userId = "default"` default is introduced.
- [ ] Manual smoke: `pnpm dev` still works after the change (with [AUTH] ticket's `SLOTHING_ALLOW_UNAUTHED_DEV=1` providing the `"default"` userId at the auth layer, not the DB layer).
