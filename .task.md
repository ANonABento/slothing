# [ATS] Dropzone silently rejects oversized/wrong-type files — no UI feedback

## Problem

In `apps/web/src/components/ats/scanner-form.tsx`, the `useDropzone` config (lines 249-258) sets `maxSize: MAX_UPLOAD_SIZE` (5MB) and `accept: { "application/pdf": [".pdf"], "text/plain": [".txt"] }`, but defines **no** `onDropRejected` callback. When a user drops a 10MB PDF or a `.docx`, react-dropzone silently ignores the file: `onDrop` is not invoked (the file is in `fileRejections`, not `acceptedFiles`), and the UI shows nothing. The user clicks the dropzone again, drops the same file, and the same thing happens — there is no error message, no toast, no anything.

`isDragReject` is wired (line 440) and turns the border red **while dragging** a bad file, but once dropped, that state clears and no persistent feedback remains.

## Repro

1. `pnpm dev`, open `/en/ats-scanner`
2. Drag a 6MB PDF (or a `.docx`) onto the dropzone
3. Drop it — observe: nothing happens. No `parseError`, no toast, no console message.
4. Inspect: `acceptedFiles` is `[]`, `fileRejections` is `[{ file, errors: [{code: "file-too-large", message: "..."}] }]` — but the form discards it.

## Root cause

`useDropzone({ onDrop, accept, maxFiles: 1, maxSize })` only invokes `onDrop` for accepted files. Rejected files are surfaced via the `onDropRejected` callback (or the returned `fileRejections` array), which the component ignores entirely.

## Fix

1. Add `onDropRejected` to the `useDropzone` config:
   ```ts
   onDropRejected: (rejections) => {
     const rej = rejections[0];
     if (!rej) return;
     const err = rej.errors[0];
     const msg =
       err?.code === "file-too-large"
         ? `File too large. Maximum size is ${formatFileSize(MAX_UPLOAD_SIZE)}.`
         : err?.code === "file-invalid-type"
           ? "Upload a PDF or TXT file."
           : err?.message || "Could not accept that file.";
     setParseError(msg);
     setUploadedFile(null);
     setParsedProfile(null);
   }
   ```
2. Wrap the `useDropzone` call so the `formatFileSize`/`MAX_UPLOAD_SIZE` are in scope (they are, at module level).
3. Also surface specific API error `code`s from `parseFile`'s response (`file_too_large`, `password_protected`, `invalid_file_type`, `invalid_file_content`) — currently any non-2xx response collapses to `data?.error || "Could not parse that resume."`. The codes from `apps/web/src/app/api/scanner/parse-resume/route.ts` lines 56-107 are designed to map to friendlier messages.
