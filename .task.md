# [SEC] Replace raw `db.prepare(...).get/all as any` casts in queries.ts with typed row interfaces

## Problem

`apps/web/src/lib/db/queries.ts:247-264` contains 6 untyped SQL prepares feeding the **profile read path** — the hottest, most-trusted code path in the app. All rows are cast `as any` / `as any[]` and then mapped into the `Profile` shape. Drift between SQL columns and `Profile` is invisible to the compiler.

```bash
$ grep -rEn ': any\b|as any\b' apps/web/src --include='*.ts' --include='*.tsx' | grep -v '\.test\.' | wc -l
12     # 10 of 12 live in apps/web/src/lib/db/* — every one is a row cast
```

Concrete sites (all production, not test):
- `apps/web/src/lib/db/queries.ts:247` — `profileRow ... as any`
- `apps/web/src/lib/db/queries.ts:252-264` — `experiences | education | skills | projects | certifications` all `.all(userId) as any[]`
- `apps/web/src/lib/db/external-calendar-events.ts:56` — `function rowToRecord(row: any)`
- `apps/web/src/lib/db/suggested-status-updates.ts:91` — `function rowToSuggestedStatusUpdate(row: any)`
- `apps/web/src/lib/db/index.ts:45/49/53` — `args as any[]` in the libsql legacy surface (low risk, but tag for completeness)

The `queries.ts` cluster is the highest impact: `getProfile()` is called by half the API surface, and column renames in migrations would silently produce `undefined` fields with zero compile-time signal.

## Repro

`pnpm type-check` passes today even if you rename `profile.summary` column to `profile.summary_text` without updating the mapper — because `profileRow.summary` is `any.summary`.

## Root cause

Legacy libSQL surface returns `Record<string, unknown>`; rather than narrow per-table, callers escape-hatch with `as any`.

## Fix

For each prepared statement in `queries.ts:240-340` (profile + sub-tables):
1. Declare a `ProfileRow`, `ExperienceRow`, `EducationRow`, `SkillRow`, `ProjectRow`, `CertificationRow` interface matching the live SQLite schema (cross-check against `drizzle-orm/schema` if a duplicate exists).
2. Replace `as any` / `as any[]` with `as ProfileRow | undefined` / `as ExperienceRow[]`.
3. Apply the same treatment to `rowToRecord` and `rowToSuggestedStatusUpdate`.
4. Leave `apps/web/src/lib/db/index.ts` `args as any[]` for a follow-up (it is bounded by libSQL's own typing).
