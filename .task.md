# [ATS] Misleading "scored in your browser, nothing is stored" copy — resume + URL are sent to server

## Problem

The marketing page `apps/web/src/app/[locale]/(marketing)/ats-scanner/page.tsx` advertises (line 21-22):

```
title: "Free and Private",
description: "Your resume is scored in your browser and nothing is stored",
```

This is **false** for both halves:

1. **"Scored in your browser"** — The score computation IS client-side (`scanResume` runs in the form via `requestAnimationFrame`), BUT the resume PDF is uploaded to `POST /api/scanner/parse-resume` for server-side parsing (`apps/web/src/app/api/scanner/parse-resume/route.ts`). The full PDF bytes traverse the network and live in server memory long enough to be parsed.
2. **"Nothing is stored"** — Strictly true at the DB layer for this public route (no `saveScanResult` call), but the server **does log** to console on failure paths (`console.error("[scanner/parse-resume] failed", ...)`) and the file passes through rate-limit caches keyed on client IP (`apps/web/src/lib/rate-limit.ts`). The job-URL scrape path (`/api/scanner/scrape-job`) likewise logs the host on failure (`safeHost(requestedUrl)`) — i.e. data IS observed and partially stored ephemerally.

This is a privacy-claim that could cause GDPR/CCPA compliance issues if a user relies on it.

## Repro

1. Open `/en/ats-scanner`.
2. Upload any PDF, open DevTools Network tab.
3. Observe `POST /api/scanner/parse-resume` carrying the **full PDF body** (multipart form-data).
4. Read page copy claiming the resume is "scored in your browser and nothing is stored".

## Fix

Pick one — preferably option A:

**Option A (preferred): Update the copy to be accurate.**
- Change description to: `"Free to use — we don't save your resume to your account or share it."` (or similar truthful phrasing).
- Add a clarifying line near the dropzone: "Your file is processed on our servers temporarily and is not stored after the scan completes."

**Option B (more work but matches the original promise): Make the scan truly client-side.**
- Move PDF parsing to the client using `pdfjs-dist`. Existing parser in `@/lib/parsers/pdf-resume` would need to be split into a client-safe variant.
- This is significantly larger scope and out of band for this ticket — file a follow-up if chosen.

This ticket scope = Option A only.
