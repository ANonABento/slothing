# [AUTH] /api/auth/[...nextauth] returns 404 (not 503) when disabled — extension + NextAuth client get cryptic errors

## Problem

`apps/web/src/app/api/auth/[...nextauth]/route.ts:9-14` returns:

```ts
function authDisabledResponse() {
  return Response.json(
    { error: "NextAuth is disabled in local development." },
    { status: 404 },
  );
}
```

When the sign-in card calls `signIn("google", ...)`, the NextAuth client library hits `/api/auth/csrf` first. Getting a `404 { error: "NextAuth is disabled..." }` for `/api/auth/csrf` is a confusing state for the client lib — it expects either 200 with a token or a real error. The user-facing result is: clicking "Sign in with Google" spins forever (because the sign-in page redirects you away before this matters, see related ticket), but any code path that *does* reach NextAuth client gets a misleading 404 that looks like a missing route, not a configuration issue.

Also, **the response leaks implementation language**: `"NextAuth is disabled in local development."` is end-user-visible if NextAuth client ever surfaces it. A more useful response is `503 Service Unavailable` with a structured payload the frontend can branch on.

## Repro

1. `pnpm dev` with no NextAuth envs
2. `curl -s localhost:3000/api/auth/csrf | jq` → `{"error": "NextAuth is disabled in local development."}` with status 404
3. NextAuth client's behavior on 404 for CSRF: silent failure, button spins.
4. Same for `/api/auth/providers`, `/api/auth/session`, `/api/auth/signin/google`.

## Suspected cause

The disabled-state was added as a placeholder, never tested against the actual NextAuth client behavior or operator monitoring tools.

## Fix

1. Return `503 Service Unavailable` (or `501 Not Implemented`) instead of 404.
2. Response shape:
   ```json
   {
     "error": "auth_disabled",
     "message": "Authentication is not configured on this instance.",
     "missing": ["GOOGLE_CLIENT_ID", "NEXTAUTH_SECRET"],
     "docs": "https://slothing.dev/docs/self-hosting#auth"
   }
   ```
3. Compute `missing` by checking which of `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `NEXTAUTH_SECRET` are absent (do NOT log secret values — only names).
4. Add cache headers `Cache-Control: no-store` so the 503 isn't cached by anything.
5. The sign-in card (related ticket) should branch on `error === "auth_disabled"` to render the disabled-state card.
