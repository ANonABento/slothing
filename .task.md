# [SEC] CSP allows `script-src 'unsafe-inline'` in production — tighten with hash/nonce

## Problem

`apps/web/src/lib/security/headers.ts:25-29` builds the production CSP `script-src` as:

```ts
const scriptSrc = ["'self'", "'unsafe-inline'"];
if (!isProduction) {
  scriptSrc.push("'unsafe-eval'");
}
```

That is: **production CSP still allows `'unsafe-inline'` for scripts**, which defeats the primary XSS-mitigation purpose of CSP. Any reflected/stored XSS into a page that renders attacker-controlled HTML (the app has 3 `dangerouslySetInnerHTML` sites — see below) becomes script execution, with no CSP backstop.

```bash
$ grep -rEn 'dangerouslySetInnerHTML' apps/web/src
apps/web/src/components/studio/resume-preview.tsx:231  dangerouslySetInnerHTML={{ __html: html }}
apps/web/src/app/layout.tsx:26  <script dangerouslySetInnerHTML={{ __html: getThemePreloadScript() }} />
apps/web/src/app/[locale]/(marketing)/page.tsx:30  dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
```

`resume-preview.tsx` renders generated HTML — if the HTML pipeline ever fails to sanitize, CSP is currently no defense.

The author's comment acknowledges the trade-off (Next.js inline bootstrap script + theme preload), but Next 14 supports `nonce`-based CSP via `headers()` in middleware, and the only intentionally-inline script is the theme preloader.

## Repro

```bash
curl -sI http://localhost:3000/en | grep -i content-security-policy
# observe: script-src 'self' 'unsafe-inline'
```

## Fix

1. Generate a per-request nonce in `apps/web/src/middleware.ts` (`crypto.randomUUID()` or `crypto.getRandomValues`).
2. Pass the nonce on the response via a request header (or `cookies()`/`headers()`).
3. Apply `script-src 'self' 'nonce-<value>'` in `headers.ts:buildContentSecurityPolicy()` (production only; dev keeps `'unsafe-inline' 'unsafe-eval'` for HMR).
4. Attach the nonce to the theme-preload `<script>` in `apps/web/src/app/layout.tsx:26`.
5. If Next's own bootstrap inline script is the blocker, evaluate `experimental.strictNextHead` or compute its hash and add via `script-src 'sha256-...'` rather than wholesale `'unsafe-inline'`.

## Acceptance criteria

- [ ] In production build (`NODE_ENV=production`), `curl -sI` shows `script-src` containing `'nonce-...'` or a `'sha256-...'` allowlist — **no `'unsafe-inline'`**.
- [ ] Dev mode unchanged (still `'unsafe-inline' 'unsafe-eval'` for HMR).
- [ ] `apps/web/src/lib/security/headers.test.ts` is extended with a production-CSP assertion that rejects `'unsafe-inline'`.
- [ ] App renders correctly (theme preload + Next hydration both function).
