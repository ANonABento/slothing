# [i18n] Intl.NumberFormat / DateTimeFormat hardcoded to "en-US" in 17+ call sites

## Problem

Date and number formatting across the app uses literal `"en-US"` regardless of the user's locale. Japanese users see "1,234" instead of "1,234" (OK, no change), but a Hindi user sees "1,234" instead of "१,२३४" or the proper "1,234" with Devanagari numerals (if preferred), and a French user sees "1,234.56" instead of "1 234,56". Salary and date displays are particularly user-visible.

next-intl actually provides `useFormatter()` (client) and `getFormatter()` (server) that auto-pick up the active locale, but none of these sites use it.

## Root cause

These call sites were written before the next-intl migration and never updated. The `apps/web/src/lib/utils.ts:39` and `upload-conflict.ts:25` are particularly bad because they're shared utilities — every caller inherits en-US.

## Fix

Two-pronged approach:

### A. Client components: use `useFormatter()`

```ts
// Before
const salaryFormatter = new Intl.NumberFormat("en-US", { style: "currency", currency: "USD" });

// After
import { useFormatter } from "next-intl";
const format = useFormatter();
format.number(value, { style: "currency", currency: "USD" });
```

Files to migrate:
- `apps/web/src/components/research/company-enrichment-dossier.tsx`
- `apps/web/src/components/opportunities/review-queue.tsx`
- `apps/web/src/components/settings/eval-health-section.tsx`
- `apps/web/src/components/jobs/job-kanban-utils.ts` (if called from client — verify)

### B. Server/util modules: accept `locale` arg

For files that aren't React components, accept the locale via parameter and call `getFormatter({ locale })` or pass the locale through to `Intl.*`:

- `apps/web/src/lib/salary/calculator.ts:236` — add `locale` param to the function signature, default `"en-US"` for backwards compat
- `apps/web/src/lib/analytics/time-series.ts:73,80,82` — same
- `apps/web/src/app/[locale]/(app)/opportunities/utils.ts:566,725,731` — these are inside `[locale]` route, so they have access to the param via the calling page; thread it through
- `apps/web/src/lib/utils.ts:39` (`formatDate`) — accept optional locale arg
- `apps/web/src/lib/upload-conflict.ts:25` — same

### C. Fix the obviously broken locale state

`apps/web/src/components/settings/locale-section.tsx:23` has `useState("en-US")` — the initial locale should come from `useLocale()`, not be hardcoded. Wire it up properly.

### D. Currency awareness (bonus)

Salary formatters always use `currency: "USD"`. Consider a separate ticket for currency-by-locale (INR for hi, JPY for ja, BRL for pt-BR, etc.) — out of scope here, but flag it in code comments.
