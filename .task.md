# [ATS] Scan button enabled for near-empty parsed PDFs — produces meaningless F-grade output

## Problem

In `apps/web/src/components/ats/scanner-form.tsx`, the `canAnalyze` guard (line 358-361) only enforces a minimum content threshold for **pasted** text:

```ts
const canAnalyze =
  Boolean(parsedProfile || resumeText.trim().length >= MIN_RESUME_LENGTH) &&
  !analyzing && !parsing;
```

When `parsedProfile` is truthy from an upload — even if the PDF parse returned a profile with **0 experiences, 0 skills, no email, no summary** — the button is enabled. The scanner then runs `scanResume()` against that empty profile, producing a deterministic "F" grade (overall ~20-30) with confusing recommendations like "Add work experience", "Add skills section", "Add email" — but the user thinks the scanner is broken because their actual resume has all those fields; it's just that the PDF parser failed to extract them.

The API path in `apps/web/src/app/api/scanner/parse-resume/route.ts` returns the parse with `confidence` and `warnings`, but the form discards these except to display "Parsed N sections, confidence X%". Low confidence does **not** block the analyze button.

## Repro

1. Open `/en/ats-scanner`.
2. Upload an image-only / scanned-image PDF (no embedded text) under 5MB.
3. The parse "succeeds" with `confidence ≈ 0.0`, `sectionsDetected: []`.
4. The success banner shows "Parsed 0 sections, confidence 0%".
5. **Scan Resume button is enabled.** Click it.
6. Result: an "F" grade with a wall of false-positive issues (missing experience, missing skills, missing email, etc.).

## Root cause

`parsedProfile` is set from `completeProfile(data.profile, rawText)` which fills in empty defaults (`experiences: []`, `skills: []`, etc.) regardless of parse quality. The `canAnalyze` guard treats "have a profile object" as "have analyzable content".

## Fix

1. Define a `hasMinimumContent(profile, fileMeta)` helper that returns false when ALL of:
   - `profile.experiences.length === 0`
   - `profile.skills.length === 0`
   - `(profile.summary?.length ?? 0) < 50`
   - OR `fileMeta?.parseConfidence < 0.25`
   - OR `fileMeta?.sectionsDetected.length === 0`
2. When `parsedProfile` is set but `hasMinimumContent` is false:
   - Disable the Scan Resume button.
   - Show a guidance message: "We couldn't extract enough content from this PDF. Try a text-based PDF, or paste your resume text below." with a "Paste text instead" CTA (already exists in the UI via `setShowPasteResume`).
3. Update `parseMessage` to also surface parser warnings (`data.warnings`) — currently they're stored in `fileMeta.warnings` but never rendered.

## Acceptance criteria

- [ ] Uploading an image-only/empty PDF leaves the Scan Resume button **disabled**.
- [ ] A clear UI message tells the user to paste manually or try a text-based PDF.
- [ ] Uploading a normal PDF with extractable content still enables the button as today.
- [ ] Pasted text under 50 chars still disables the button (existing behavior preserved).
- [ ] Existing `scanner-form.test.tsx` tests pass.
- [ ] New test: empty-profile parse keeps Scan Resume disabled.
