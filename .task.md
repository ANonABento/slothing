# [DASH] Dashboard fires 5 unconditional `/api` calls in one Promise.all — no retries, no per-call error isolation

## Problem

`apps/web/src/app/[locale]/(app)/dashboard/page.tsx:117-172` does:

```ts
const [profileData, documentsData, analyticsData, onboardingData, streakData] =
  await Promise.all([
    fetchJson("/api/profile"),
    fetchJson("/api/documents?limit=200"),
    fetchJson("/api/analytics"),
    fetchJson("/api/onboarding/dismiss"),
    fetchJson("/api/streak"),
  ]);
```

Three real UX problems:

1. **All-or-nothing failure.** If `/api/streak` returns 500 (e.g. streak table missing on a fresh install), the entire dashboard renders the error state — even though profile, documents, analytics, and onboarding all came back fine. Users see "Could not load dashboard" when 80% of the data is available.

2. **No retries, no backoff.** A transient 502 from any one endpoint = full error page. No `retry` button on the error card (only "Try again" which is a hard `window.location.reload()` — wipes any client state).

3. **`/api/onboarding/dismiss` shape is overloaded.** Same endpoint is used to GET the dismissed state and POST to dismiss it. This dual-purpose route forces the dashboard to fetch a "control" endpoint at mount just to know the user's onboarding cohort. Should be `/api/onboarding/state` (GET) + `/api/onboarding/dismiss` (POST).

Also: the `fetchJson` helper throws `Error("AUTH_REQUIRED")` for 401, but the auth-disabled dev mode never returns 401 (see related `[AUTH]` tickets) — so this branch is currently dead code, masking the fact that the bypass is active.

## Repro

1. `pnpm dev` with `NEXTAUTH_*` envs unset (current state).
2. In Chrome DevTools → Network → right-click `/api/streak` → "Block request URL".
3. Reload `/en/dashboard`.
4. Observe: full-page error "Could not load data", even though `/api/profile` (which controls the "Welcome back" header + completeness) returned 200.

## Suspected cause

`Promise.all` was chosen for simple parallelism but maps the failure model wrong for a dashboard, which is fundamentally a collage of independent widgets.

## Fix

1. Replace `Promise.all` with `Promise.allSettled`. Each panel takes its own `{ data, error }` state. Errors render an inline "Couldn't load this section · Retry" inside the panel, not a full-page takeover.
2. Add a per-fetch `retry(times: 2, backoffMs: 300)` helper for transient 5xx.
3. Promote the 401 branch: when *any* request returns 401, redirect to `/sign-in?callbackUrl=/dashboard` instead of rendering an inline "sign in" card (matches `(app)/layout.tsx`'s server-side gate).
4. Split `/api/onboarding/dismiss` GET out into `/api/onboarding/state` (mirror existing pattern: `/api/streak` is GET, `/api/streak/dismiss` could be POST). Keep `dismiss` as POST-only.
5. Keep the existing `DashboardSkeleton` for the first paint, then transition individual `<Suspense>` panels as their data arrives.

## Acceptance criteria

- [ ] Blocking `/api/streak` in devtools does NOT prevent the dashboard from rendering profile + analytics + recent jobs.
- [ ] Each panel that fails shows an inline error with a `Retry` button that only re-fetches that panel.
- [ ] 401 on any call redirects to sign-in with proper `callbackUrl`.
- [ ] `/api/onboarding/state` (GET) returns `{ dismissedAt, firstName, ... }`; `/api/onboarding/dismiss` (POST) only dismisses.
- [ ] Dashboard test in `dashboard/page.test.tsx` covers the partial-failure path.
- [ ] No new console.error in happy path.
