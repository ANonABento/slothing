# [i18n] translate:messages needs CI guardrail â€” drift detection + auto-PR on en.json change

## Problem

`pnpm translate:messages` is a manual step. When a contributor adds a key to `apps/web/src/messages/en.json`, nothing forces them to fan-out translations. Result: silent regressions where Japanese/Hindi users see English fallbacks (or `MISSING_MESSAGE` errors from next-intl, depending on config).

Today the script has not been re-run since `2026-05-10T09:48:08.069Z` (per `_meta.generatedAt` in es.json). Any en.json key added in the last 24+ hours could already be drifting.

## Repro

```sh
cd /home/anonabento/slothing/apps/web
python3 -c "
import json
for f in ['en','es','fr','ja','ko','hi','pt-BR','zh-CN']:
    d = json.load(open(f'src/messages/{f}.json'))
    print(f'{f}: meta={d.get(\"_meta\",{}).get(\"generatedAt\",\"none\")}')
"
```

Output today (2026-05-11):
```
en: meta=none                                  # en.json has no _meta; this is by design
es: meta=2026-05-10T09:48:08.069Z              # 1 day old
zh-CN: meta=2026-05-10T09:48:59.697Z
pt-BR: meta=2026-05-10T09:49:59.153Z
hi: meta=2026-05-10T09:51:09.616Z
fr: meta=2026-05-10T09:52:09.947Z

ko: meta=2026-05-10T09:54:16.606Z
```

If `en.json` was committed today with a new key, all 7 locales are stale. CI doesn't catch this.

## Root cause

`apps/web/scripts/translate-messages.ts` is invoked only on-demand. There's no CI hook, no pre-commit, no lint that flags missing keys.

## Fix

Three-layer guardrail. Pick at least Layer 1 + Layer 2 in this ticket; Layer 3 is optional.

### Layer 1: CI drift check (required)

Add a script `apps/web/scripts/check-translations.ts` that:
1. Loads `en.json` and flattens all keys.
2. For each non-en locale, flattens and computes:
   - **Missing keys:** present in en but absent in locale.
   - **Extra keys:** present in locale but absent in en (other than `_meta.*`).
   - **Identical-to-en suspicious:** values that match en verbatim, excluding `passthroughPaths` and `brandTerms` from `translate-messages.ts`.
3. Exits non-zero if missing > 0 or extras > 0. Identical-to-en is a warning (not failure) since some terms are legitimately loanwords.
4. Outputs a markdown table to stderr for CI logs.

Wire it into:
- `apps/web/package.json`: `"check:translations": "tsx scripts/check-translations.ts"`
- Existing GitHub Actions workflow under `.github/workflows/` â€” add a step that runs after `pnpm install`.
- `pnpm test` script (so local dev also catches it).

### Layer 2: Auto-fanout on en.json change (recommended)

Add a GitHub Action `.github/workflows/i18n-fanout.yml` that:
- Triggers on `push` / `pull_request` when `apps/web/src/messages/en.json` changed.
- Runs `pnpm translate:messages` (with `ANTHROPIC_API_KEY` from secrets).
- If any non-en JSON changed, opens an auto-PR labeled `i18n-fanout` with the diff (or commits directly to the PR branch if the trigger is a PR).
- Skips if commit message contains `[skip-i18n]`.

### Layer 3 (optional): Pre-commit hook

`.husky/pre-commit` (or lefthook config) â€” if `en.json` is staged, prompt the dev to run `pnpm translate:messages` or warn.
