# [i18n] Marketing pages have no locale switcher — users stuck in detected locale

## Problem

The marketing navbar (`apps/web/src/app/[locale]/(marketing)/components/navbar.tsx`) has **no language switcher**. Once `localeDetection: true` (middleware) routes a user to `/fr` or `/ja` based on their browser `Accept-Language`, they cannot switch back to English (or any other locale) from any public page — no UI for it.

The only language switcher lives in `apps/web/src/components/settings/language-section.tsx`, which is **behind auth** in `/settings/locale`.

## Repro

```sh
grep -rn "useLocale\|LanguageSection\|locale.*switch\|LocaleSwitcher" apps/web/src/app/\[locale\]/\(marketing\) --include="*.tsx" 2>/dev/null
```
Returns only `navbar.tsx:4: import { useLocale } from "next-intl";` — used to build the dashboard callback URL, not to render a switcher.

Concrete failure path:
1. French-locale browser hits `slothing.work` → middleware redirects to `/fr`
2. User wants the English version (better keyword recall, friend shared link as English copy)
3. No UI on `/fr`, `/fr/pricing`, `/fr/ats-scanner`, `/fr/extension` to switch
4. User has to manually edit URL or clear cookies — most won't

## Root cause

`LanguageSection` was built for the authenticated `/settings` page only. The marketing nav was never updated to include a switcher.

## Fix

1. Extract the locale-switch logic from `apps/web/src/components/settings/language-section.tsx` into a reusable `apps/web/src/components/i18n/locale-switcher.tsx` component with two variants:
   - **`<LocaleSwitcherDropdown />`** — full select-based UI (current settings behavior)
   - **`<LocaleSwitcherCompact />`** — globe icon + minimal menu, suitable for nav bars
2. Mount `<LocaleSwitcherCompact />` in the marketing navbar (`apps/web/src/app/[locale]/(marketing)/components/navbar.tsx`), both desktop and mobile menu.
3. Use `usePathname()` + `useRouter()` from `@/i18n/navigation` (the next-intl wrappers) so the switch preserves the current path across locales (`/fr/pricing` → `/ja/pricing`, not `/ja`).
4. Set `NEXT_LOCALE` cookie on switch (same as `language-section.tsx:23`) so subsequent visits remember the choice.
5. Add `aria-label` (translated) and `lang` attribute on each option (`<option lang="ja">日本語</option>`) so screen readers announce the language correctly.
6. Add `marketing.nav.languageSwitcher.{label,description}` keys to `en.json` + re-run `pnpm translate:messages`.

## Acceptance criteria

- [ ] Marketing navbar shows a globe/language icon on all `(marketing)` pages
- [ ] Clicking it shows all 8 locales (use the existing `locales` array from `@/i18n`)
- [ ] Switching from `/fr/pricing` → French select → Japanese → lands on `/ja/pricing` (path preserved)
- [ ] `NEXT_LOCALE` cookie updates on switch
- [ ] Mobile menu also includes the switcher
- [ ] Keyboard navigation works (Tab to switcher, Enter/Space opens, Arrow keys cycle, Esc closes)
- [ ] `pnpm test` and `pnpm type-check` pass
