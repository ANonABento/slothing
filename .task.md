# [AUTH] requireAuth() silently authenticates everyone as "default" user when NextAuth is disabled

## Problem

When NextAuth is not configured (missing `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` / `NEXTAUTH_SECRET`), `requireAuth()` in `apps/web/src/lib/auth.ts:42-53` returns `{ userId: "default" }` for ANY request â€” no headers, no cookies, no nothing. Every "protected" API route (`/api/profile`, `/api/jobs/*`, `/api/analytics`, `/api/onboarding/*`, `/api/streak`, `/api/documents`, etc.) then happily returns or mutates data belonging to user `"default"`.

Combined with `apps/web/src/app/[locale]/(app)/layout.tsx:53-55` â€” which short-circuits the session check with `if (!isNextAuthConfigured()) return <AppShell>{children}</AppShell>;` â€” the **entire authenticated surface (`/en/dashboard`, `/en/profile`, `/en/opportunities`, `/en/bank`, etc.) is publicly accessible** the moment any one of those three env vars is missing.

This is intended for "local dev convenience" but the failure mode is silent: a misconfigured prod deploy (forgot `NEXTAUTH_SECRET`?) ships as a wide-open multi-tenant DB read/write for the `"default"` row.

## Repro

1. Unset `GOOGLE_CLIENT_ID` (or `GOOGLE_CLIENT_SECRET` or `NEXTAUTH_SECRET`).
2. `pnpm dev`
3. `curl localhost:3000/api/profile` â€” returns 200 with whatever `"default"` user's profile is.
4. `curl -X PUT localhost:3000/api/profile -H 'content-type: application/json' -d '{"name":"hacker"}'` â€” accepted, writes to `"default"` user row.
5. Open `/en/dashboard` in any browser, no sign-in needed â€” full app shell loads.

## Root cause

Two over-eager dev shortcuts compound:
- `isNextAuthConfigured()` returns false â†’ `requireAuth()` skips the session check and returns a hardcoded user id (`apps/web/src/lib/auth.ts:32, 43`).
- The `(app)` layout uses the same gate (`apps/web/src/app/[locale]/(app)/layout.tsx:53-55`).

There is no signal â€” no log line, no `X-Dev-Auth-Bypass` response header â€” telling the operator that auth is bypassed.

## Fix

1. Gate the "local dev user" fallback behind an explicit env var, e.g. `SLOTHING_ALLOW_UNAUTHED_DEV=1`, **not** the absence of NextAuth env vars.
2. When the fallback fires, log a single bright warning on first use per process and add a response header (`X-Slothing-Dev-Auth: default-user`) so it's obvious in browser devtools / curl.
3. In production builds (`NODE_ENV === "production"`), refuse to start (or return 503 from `requireAuth()`) if `isNextAuthConfigured()` is false. Failing closed > silently shipping a public DB.
4. Update `(app)/layout.tsx` to mirror the same gate â€” i.e. only render `<AppShell>` without a session when `SLOTHING_ALLOW_UNAUTHED_DEV=1`, otherwise redirect to a "Sign-in unavailable" page.
